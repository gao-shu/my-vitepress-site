# 01 本地部署：Ollama / vLLM / LM Studio 怎么选，怎么跑

> 目标：用最小成本把模型在本地跑起来，优先满足“写代码/写文档/本地知识库问答”的日常需求。

## 1）本地部署适合什么场景

- **隐私优先**：不希望把笔记/代码发到云端
- **弱网/断网也能用**
- **成本可控**：少量调用、长期使用
- **延迟可控**：同机推理通常更稳定

不太适合：

- 你需要很强推理/长上下文/高并发（本机会吃不消）

## 2）三种本地部署路线（怎么选）

### 2.1 Ollama（最省心）

- **适合**：快速上手、本地小模型、日常问答/总结/写作
- **优点**：安装简单、模型管理方便、生态成熟
- **缺点**：极限性能不如专业推理引擎，深度定制能力有限

### 2.2 vLLM（偏工程化/性能）

- **适合**：你想把本地模型做成“服务”，后续接 Agent/MVP Server
- **优点**：吞吐与服务化能力更强（尤其在 GPU 环境）
- **缺点**：部署与参数更工程化，上手成本更高

### 2.3 LM Studio（偏桌面端体验）

- **适合**：你希望“可视化管理模型 + 本地聊天”快速使用
- **优点**：界面友好，上手快
- **缺点**：更偏个人使用，不太像服务端组件

## 3）本地部署的关键约束（先看再装）

- **硬件**：
  - CPU 能跑，但速度会慢
  - 有 GPU（尤其显存大）体验更好
- **模型大小**：
  - 先从“小模型”开始，能跑通闭环比追求大模型更重要
- **上下文长度**：
  - 知识库问答/长文摘要更吃上下文与显存

## 4）本地使用的推荐姿势（配合你的知识库）

- **个人学习/面试**：
  - 先本地跑一个可用模型
  - 用你阶段一/二的脚本，把 `docs/` 里的内容喂给模型做总结/出题
- **RAG/Agent**（后续）：
  - 本地模型作为推理层
  - `MVP Server` 作为“上下文提供层”

## 5）最小验收（今天就能完成）

- 你能在本地跑通一次：输入一段 Markdown → 输出固定模板总结
- 你能记录下 3 个指标：
  - 速度（体感/耗时）
  - 稳定性（是否容易崩/卡）
  - 成本（本地=硬件成本 vs 云端=调用成本）
