# 02 云端部署：托管 API / 自建推理 / 混合方案

> 目标：根据预算与稳定性要求，选择“直接用云 API”还是“自建推理服务”，并能接入你的 Agent/MVP Server。

## 1）云端适合什么场景

- **要强能力**：更强推理、更长上下文、更好代码能力
- **要稳定**：不折腾本地环境，不受个人机器影响
- **要并发**：团队/小产品多人使用
- **要快速上线**：先跑通业务闭环，再考虑成本优化

## 2）三种云端路线（怎么选）

### 2.1 托管 API（最快）

- **适合**：你要最快把功能做出来（脚本/小工具/原型）
- **优点**：稳定、维护成本低、迭代快
- **缺点**：长期成本与合规/隐私要评估

### 2.2 自建推理（GPU 实例 + 推理引擎）

- **适合**：你要“可控、可定制、可成本优化”，并有一定运维能力
- **优点**：模型/版本/日志更可控
- **缺点**：搭建与维护成本更高（监控、扩容、限流等）

### 2.3 混合方案（推荐）

- **策略**：
  - 日常/隐私：本地小模型
  - 关键任务/高质量：云端强模型
- **收益**：
  - 成本可控
  - 质量可控
  - 体验稳定

## 3）和你的工程怎么对接（建议架构）

- 你的应用（Agent/前端）不直接绑死某一家模型
- 中间加一层 `LLM Gateway`（可以先写在你的服务里）：
  - 统一请求格式
  - 统一日志/限流/重试
  - 可切换：本地推理 / 云端 API

## 4）最小验收

- 你能用云端 API 做一次“面试题生成器”
- 并且把调用封装成一个函数/接口，后续可替换模型提供方
